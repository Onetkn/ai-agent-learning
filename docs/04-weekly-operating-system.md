# 每周推进系统

你每周只有 6-8 小时，所以关键不是“塞更多内容”，而是固定节奏、稳定产出。

## 默认周节奏

### 方案 A：6 小时版

- 2 小时：读官方文档或源码
- 2 小时：写代码或搭实验
- 1 小时：做测试或对比评测
- 1 小时：写周复盘

### 方案 B：8 小时版

- 2 小时：读资料
- 3 小时：实现项目
- 2 小时：评测与调试
- 1 小时：复盘与下周计划

## 每周 Definition of Done

一周结束前，至少留下 3 类产物中的 2 类：

- 笔记
- 代码/实验
- 复盘/报告

最好全部都有。

## 每周固定动作

### 周开始

- 明确本周只攻一个主主题
- 从 `docs/01-roadmap.md` 里选出 1 个交付物
- 提前定义完成标准

### 周中

- 记录至少 3 个问题
- 不懂的概念用自己的话重写
- 出现失败案例时立刻归档，不要等“全部搞懂了再写”

### 周末

- 更新一篇周复盘
- 记录：
  - 本周最有效的方法
  - 最大误区
  - 下周要删掉的分心项

## 学习文件使用建议

- `notes/`：放阅读笔记、概念解释、框架对比
- `experiments/`：放 prompt、RAG、agent 参数对比记录
- `reports/`：放正式实验报告和阶段总结
- `projects/`：放每个阶段的实现与 README

## 推荐命名

- `notes/2026-04-llm-basics.md`
- `experiments/2026-05-rag-rerank-comparison.md`
- `reports/phase-3-rag-eval-report.md`
- `projects/project-03-research-agent/README.md`

## 每次实验都要回答的 4 个问题

- 我改了什么
- 为什么改
- 指标怎么变了
- 这个改动是否值得保留

## 什么时候允许切换主题

只有满足下面任一条件才切：

- 当前主题已有明确交付物
- 当前主题连续卡住，但已完成失败记录
- 新主题是旧主题的前置依赖

否则继续深挖，不要被新框架带节奏。
