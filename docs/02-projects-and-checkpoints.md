# 项目梯度与验收清单

这份文档把“学什么”变成“交付什么”。每个项目都要有输入、输出、日志、失败分析和验收标准。

## 项目 1：结构化抽取器

目标：掌握结构化输出、schema 设计、基本评测。

### 最小功能

- 输入一段非结构化文本
- 输出结构化 JSON
- 对字段缺失、格式错误给出显式处理

### 必须交付

- 清晰的输入/输出 schema
- 至少 10 条测试样本
- 至少 3 条失败案例
- 一份字段设计说明

### 通过标准

- 主要字段稳定输出
- 模型输出异常时能被解析和修复或显式报错
- 你能解释每个字段为什么存在

## 项目 2：工具型 FAQ/助手

目标：掌握 tool calling、参数验证、错误处理。

### 最小功能

- 至少接 2-3 个工具
- 工具有明确描述、参数和错误返回
- agent 能在有限步数内完成任务

### 必须交付

- 工具列表与 schema
- 3 类错误场景
- stop condition 说明
- 一份工具选择失败分析

### 通过标准

- 工具不会被无限循环调用
- 错误参数会被拦截
- 你能解释“为什么这个系统需要 agent，而不是纯流程”

## 项目 3：研究助理 agent

目标：掌握单 agent 工作流、引用、失败分类。

### 最小功能

- 搜索网页或本地资料
- 抽取事实
- 给出来源引用
- 子任务数量受控

### 必须交付

- 一组代表性问题集
- 至少 1 次 trace 分析
- 失败分类表
- 引用输出格式说明

### 通过标准

- 回答中能区分事实、推断和缺失信息
- 引用可追溯
- 能总结最常见失败模式

## 项目 4：私有知识库问答系统

目标：掌握 RAG、检索对比实验和基础 eval。

### 最小功能

- 文档导入
- chunk/embedding/retrieve
- 输出引用
- 至少 3 个版本对比

### 必须交付

- 数据集来源说明
- chunk 策略说明
- baseline / rerank / query rewrite 三组结果
- 一份实验报告

### 通过标准

- 不是只展示“最好样例”，而是看整组样本
- 你能指出哪个版本在什么任务上更好
- 你能说明代价：速度、成本、复杂度

## 项目 5：MCP Server + Guardrails Workflow

目标：掌握协议化工具接入、最小生产思维和安全边界。

### 最小功能

- 一个 MCP server
- 2-3 个工具
- 1 个资源
- 人工确认节点

### 必须交付

- 工具 schema
- 资源说明
- 错误码或错误信息策略
- 权限边界说明
- 风险操作清单

### 通过标准

- 工具职责清晰，不是“大而全万能函数”
- 危险操作有确认机制
- 你能解释 MCP 比手写胶水代码的收益

## 项目 6：前沿增强实验

从下面四项选两项：

- coding agent
- browser/computer-use
- multi-agent
- DSPy

### 每个方向都必须回答

- 它解决了单 agent 解决不好的什么问题
- 它引入了什么新复杂度
- 它在什么条件下不值得用

### 通过标准

- 你做的是对比实验，不是概念搬运
- 你能给出清晰的使用边界

## 最终作品集要求

你至少要公开一个完整作品，候选方向：

- `RAG + eval + trace + MCP`
- `coding agent` 可复现实验
- `browser/computer-use` 可复现实验

### 最终作品必须包含

- 问题定义
- 系统架构图
- 数据流说明
- 核心实验结果
- 失败案例
- 适用边界

## 总体验收标准

当你满足以下条件，这条路线就算真的落地了：

- 你能不用框架黑盒解释 tool calling、RAG、memory、eval、MCP
- 你能用数据比较两个版本，而不是靠主观印象
- 你能给一个系统设计明确 stop condition、权限边界和失败恢复
- 你能区分 demo、实验、工程化三种不同层级
