# 第 2 周笔记：结构化抽取器最小设计

更新日期：2026-04-20

## 本周目标

第 2 周的主线不是“做一个很大的 agent”，而是先把一个可验证、可调试、可失败归档的结构化抽取器做稳。按照路线图，本周至少要落下三类核心材料：

- 输入/输出 schema
- 10 条测试样本
- 3 条失败案例

这三类材料比“先写很多代码”更重要，因为它们决定了后面做 prompt、重试、校验时有没有共同基线。

## 这个项目到底要解决什么问题

结构化抽取器的目标是：把一段自然语言文本，稳定转换成固定字段的 JSON。它最适合处理这类任务：

- 工单抽取
- 客服留言归档
- 表单补录
- 会议纪要字段整理

它不应该承担的职责：

- 自由问答
- 多轮规划
- 开放式搜索
- 带复杂工具调用的代理决策

换句话说，本周重点是“固定 schema 的抽取系统”，不是通用 agent。

## 最小系统设计

本周最小实现建议拆成 5 层：

1. 原始输入
   接收一段待抽取文本，不在入口做复杂预处理。
2. 抽取提示词
   明确字段、允许值、缺失策略、禁止猜测。
3. 结构约束
   用 JSON Schema 或 Pydantic 模型约束输出。
4. 结果校验
   发现缺字段、枚举值错误、类型错误时立即标记失败。
5. 失败归档
   保留原文、输出、失败原因，方便后续补 few-shot 或改 schema。

## 本周建议先学什么

围绕这 4 个主题就够了：

- Python `async`：先理解为什么后续调用模型或 HTTP API 时会涉及并发与超时控制
- HTTP API：知道请求、响应、状态码、超时、重试的最小概念
- Pydantic：把“输出结构”变成可验证对象
- 日志：至少区分成功、失败、校验失败三类记录

- Python `logging` 文档

  https://developers.openai.com/api/docs/

读资料时至少回答 5 个问题：

- schema 为什么比“在 prompt 里口头要求 JSON”更稳
- 哪些字段必须枚举，哪些字段允许 `null`
- 校验失败后应该重试，还是直接归档
- 日志里最少要记哪些字段
- 什么情况下不该继续扩展成 agent

## 为什么先做 schema，再做 prompt 微调

如果没有 schema，你很难判断一次输出失败，到底是：

- 模型理解错了
- 字段定义含糊
- 枚举设计太随意
- 输出校验规则没写清

先固定 schema 的好处是：失败会更容易定位。第 2 周最值得练习的不是“写出一个最强 prompt”，而是把系统边界说清楚。

## 本周完成标准

满足下面 4 条，就算本周交付达标：

- 有一份明确的输入/输出 schema
- 有 10 条覆盖不同情况的测试样本
- 有至少 3 条失败案例，并写明失败原因
- 能解释为什么这些字段这样设计

## 我对第 2 周的保守实现假设

由于仓库里目前还没有真实的抽取器代码，本周先补“可检索的设计与测试资产”，不贸然编造 API、脚本入口或配置。这样做的好处是：

- diff 小，容易审阅
- 不会覆盖用户已有实现
- 后续写代码时可以直接拿这些文件做输入基线
