## 结构化抽取器到底要解决什么问题

结构化抽取器的目标是：把一段自然语言文本，稳定转换成固定字段的 JSON。它最适合处理的任务是：

- 工单抽取
- 客服留言归档
- 表单补录
- 会议纪要字段整理

它不应该承担的职责是：

- 自由问答
- 开放式网页搜索
- 多轮计划拆解
- 带复杂工具选择的代理决策

一句话版本：

- 这周要做的是“固定 schema 的抽取系统”。
- 不是“能做很多事的通用 agent”。

## 结构化抽取器、workflow、agent 的边界

| 概念 | 它是什么 | 解决的问题 | 什么时候用 | 容易误解的地方 |
| --- | --- | --- | --- | --- |
| 结构化抽取器 | 把非结构化文本转成固定字段输出的组件 | 让后续程序可以稳定消费结果 | 分类、抽取、工单归档、字段补录 | 它不是问答助手，也不是规划器 |
| workflow | 一组稳定、可重复的步骤编排 | 把输入、抽取、校验、重试、归档串起来 | 先抽取再校验、失败再记录、人工复核 | workflow 不等于 agent |
| agent | 能基于上下文自主决定下一步动作的系统 | 处理开放问题和不确定路径 | 搜索、工具选择、多步任务推进 | 很多固定抽取任务其实不需要 agent |

一句话判断：

- 如果字段固定、输出固定、验收标准固定，优先做抽取器 + workflow。
- 不要因为“LLM 项目”就默认上 agent。

## 为什么第 2 周先做 schema，而不是先堆 prompt

第 2 周最容易犯的错误是：一上来就改 prompt，看到输出不稳就继续加样例、加规则、加例外，最后自己也不知道问题到底出在哪里。

先做 schema 的意义是：

- 先把“什么叫正确输出”讲清楚。
- 先把“哪些字段允许缺失”讲清楚。
- 先把“哪些值必须落在枚举内”讲清楚。
- 先把“哪些失败应该归档而不是糊过去”讲清楚。

没有 schema 时，一次失败很难定位到底是：

- 模型理解错了
- 字段定义含糊
- 输出格式不稳定
- 枚举边界没说清
- 校验逻辑没写出来

参考：Pydantic 文档，2026-04-20 查验。
我的理解是：schema 的核心价值不是“好看”，而是把输出变成可以验证、可以拦截、可以复盘的对象。

## 输入/输出 schema 应该怎么想

本周最小输入形态可以很简单：

```json
{
  "ticket_text": "string"
}
```

输出形态也应该尽量固定：

```json
{
  "customer_name": "string | null",
  "product": "string | null",
  "issue_type": "string | null",
  "urgency": "high | medium | low | null"
}
```

这里最关键的不是字段数量，而是 3 个边界：

1. 缺失字段要不要允许 `null`
2. 枚举字段是不是只能输出固定值
3. 允许不允许模型“合理猜测”

第 2 周的默认答案应该是：

- 允许 `null`
- 枚举必须固定
- 不允许猜测

因为你当前最需要的是可控性，而不是覆盖一切模糊情况。

## 为什么 10 条测试样本和 3 条失败案例比“多写代码”更重要

路线图里要求你至少留下：

- 输入/输出 schema
- 10 条测试样本
- 3 条失败案例

这不是为了凑数，而是因为这三类资产刚好对应三个层面：

- schema 负责定义对错
- 测试样本负责提供基线
- 失败案例负责告诉你系统会在哪里失控

如果没有测试样本，你很容易出现“今天感觉效果不错，明天换一句输入又坏了”。
如果没有失败案例，你后面做 retry、few-shot、日志归档时就没有真实抓手。

所以第 2 周最重要的不是“把项目做大”，而是先把验收面做实。

## Python async、HTTP API、日志在这周分别起什么作用

### 1. Python `async`

这周不用把 `async` 学成并发专家，但至少要知道：模型调用、HTTP 请求、超时控制、批量测试，后面都可能涉及异步执行。
参考：Python `asyncio` 文档，2026-04-20 查验。

你当前需要先理解：

- 什么是协程
- 什么是 `await`
- 为什么 I/O 等待时异步更自然

### 2. HTTP API

因为你后面无论接 OpenAI 还是别的模型服务，本质上都在做请求/响应。
你至少要知道：

- 请求体长什么样
- 响应体长什么样
- 超时和失败怎么区分
- 什么情况适合重试，什么情况不该重试

### 3. 日志

日志不是“后面再补”的东西。对抽取器来说，最少要记下：

- 原始输入
- 模型输出
- 校验结果
- 失败原因标签

参考：Python `logging` 文档，2026-04-20 查验。
我的理解是：第 2 周的日志不是为了观测大规模生产流量，而是为了让你别在失败后只能靠回忆排查。

## 第 2 周一个稳妥的最小 workflow

先不要做复杂化，最小 workflow 足够了：

1. 接收一段原始文本
2. 调用模型执行抽取
3. 按 schema 校验输出
4. 成功则保存结果
5. 失败则记录失败原因和原始样本

这个 workflow 的关键不在“步数多”，而在每一步都有明确职责：

- 模型负责抽取
- schema 负责约束
- 校验负责拦截
- 日志负责复盘

只要这四件事分清楚，你后面再加 retry 或 few-shot 才不会乱。

## 这一周最重要的 6 个判断

1. 结构化抽取器优先解决“稳定输出”，不是“能力炫技”。
2. schema 先于 prompt 微调，因为你得先定义什么叫正确。
3. `null` 比猜测更值钱，因为它能保住系统可控性。
4. 测试样本不是附属品，而是后续评估和回归的基线。
5. 失败案例不是负资产，而是后续优化最有价值的输入。
6. 大多数固定字段抽取任务，用 workflow 就够了，不必急着上 agent。
